# AGENTS.md

Agent instructions for the Mar Sem Lixo repository. Rules here mirror `CLAUDE.md`
and `backend/CLAUDE.md`; if anything conflicts, those files win.

## Project purpose

Mar Sem Lixo is a PWA + REST API for environmental-NGO volunteers to record
beach/lagoon waste collection (Região dos Lagos, RJ), with consolidated Excel
reports for coordination.

Two roles: **VOLUNTARIO** (records waste in the field, offline-capable) and
**COORDENADOR** (manages mutirões and areas, views reports).

## Stack (backend)

- Spring Boot 3.3+, Java 21, virtual threads (`spring.threads.virtual.enabled=true`)
- Spring Data JPA + PostgreSQL with PostGIS (`postgis/postgis:16-3.4` in tests)
- Spring Security + JWT (Google OAuth on frontend; backend validates ID Token and
  issues its own JWT)
- Flyway for migrations; Springdoc OpenAPI; Apache POI for Excel
- Testcontainers + JUnit 5 + ArchUnit; no H2 ever

## Build and test commands

Always run from `backend/`. Always use `./mvnw`, never bare `mvn`.

```bash
./mvnw spring-boot:run          # start dev server on :8080
./mvnw test                     # unit + integration tests
./mvnw verify                   # full suite including ArchUnit
docker build -t mar-sem-lixo-api:dev .
```

Useful local endpoints: `/actuator/health`, `/swagger-ui.html`, `/v3/api-docs`.

## Module organization

Domain packages under `com.marsemlixo.api.<context>`:
`auth`, `area`, `mutirao`, `residuo`, `relatorio`.

Each context has its own `controller / service / repository / domain` layers.

**Cross-module access rules:**
- A module may inject another module's **repository for read-only queries**.
  This is an accepted pragmatic choice in this codebase.
- A module must **never mutate** another module's data through its repository
  directly — that must go through the owning module's service interface.
- A module must **never call** another module's `controller`.

## ID strategy

| Entity | PK type | Generated by |
|---|---|---|
| `Voluntario`, `Area`, `Mutirao`, `RefreshToken` | `BIGINT GENERATED ALWAYS AS IDENTITY` | Database |
| `RegistroResiduo` | `UUID` (no `@GeneratedValue`) | Client — required for offline idempotency |

Do not change this without an ADR.

## Java conventions

- Records for DTOs (`*Request`, `*Response`).
- Sealed types for closed hierarchies; pattern matching in switch.
- Business logic lives in services, never in controllers.
- Always separate input DTO from output DTO — never expose JPA entities directly.
- `@Valid` on every `@RequestBody` that has constraints.
- `catch (Exception e)` only with explicit justification in a comment.
- No hardcoded credentials anywhere.

## SQL conventions

- Pure PostgreSQL — no schema prefix in DDL or queries (`schema.tabela` is forbidden).
- Migrations versioned as `V<N>__description.sql` via Flyway.
- Edit migrations in-place only while the project is pre-production (no data to preserve).

## Testing conventions

- Class names: `*Test` (unit), `*IT` (integration) — Surefire picks both up.
- Integration tests use Testcontainers with a real PostgreSQL+PostGIS container, never H2.
- `@BeforeEach` must truncate all affected tables in a single `TRUNCATE TABLE a, b, c`
  statement (PostgreSQL resolves FK order internally).
- Insert fixtures via `JdbcTemplate.queryForObject("INSERT … RETURNING id", Long.class, …)`
  to capture IDENTITY-generated PKs.
- Token helpers must use `Long` subjects (numeric string), not UUID strings.

## Architectural changes

Any change that affects: authentication strategy, persistence layer, deployment
topology, domain model, inter-module communication, or API contract requires a
new ADR in `docs/architecture/adr/` **before** the code. Read existing ADRs
there before proposing a pattern that may already be decided.

## Reference documentation
